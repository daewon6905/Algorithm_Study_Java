📋 프로그래머스 - 피로도 (Java)💡 

문제 유형알고리즘: 완전 탐색, 백트래킹, 순열

시간 복잡도: $O(N!)$ (던전의 개수 $N \le 8$로 매우 작기 때문에 완전 탐색이 가능합니다.)

🛠 풀이 별 접근 방식 및 비교제시된 두 풀이는 모두 가능한 모든 던전 탐험 순서(순열)를 확인한다는 점에서는 동일하지만, 구현 방식(가지치기 유무 및 메모리 활용)에서 명확한 차이가 있습니다.
1. 피로도sol1: **백트래킹을 이용한 실시간 탐색 핵심 아이디어**: 순열을 끝까지 만들지 않고, 탐색 과정에서 즉시 피로도를 차감하며 조건을 만족할 때만 깊이 우선 탐색(DFS)을 진행합니다.동작 방식:현재 남은 피로도(k)가 던전의 최소 필요 피로도(dungeons[i][0])보다 크거나 같을 때만 visited를 true로 바꾸고 재귀 호출을 진행합니다.매 재귀 단계마다 현재까지 탐험한 던전의 수(count)를 maxDungeons와 비교하여 최댓값을 갱신합니다.장점:피로도가 부족하면 더 이상 깊이 들어가지 않으므로 불필요한 연산을 줄이는 가지치기(Pruning)가 자연스럽게 이루어집니다.순열을 저장하기 위한 별도의 리스트 공간이 필요 없어 공간 복잡도 측면에서 효율적입니다.


2. 피로도sol2: **순열(Permutation) 선생성 후 완전 탐색핵심 아이디어**: 발생 가능한 모든 던전의 방문 순서(인덱스 순열)를 미리 리스트에 모두 저장해 둔 뒤, 각 경로를 하나씩 처음부터 끝까지 시뮬레이션합니다.동작 방식:permute 함수를 통해 $N!$가지의 모든 인덱스 조합을 List<List<Integer>> P에 먼저 저장합니다.저장된 모든 경로(route)를 하나씩 꺼내어, 해당 순서대로 던전을 돌 수 있는지 루프를 돌며 체크하고 최댓값을 찾습니다.아쉬운 점:던전의 개수가 늘어날 경우 모든 순열을 메모리에 유지해야 하므로 메모리 낭비가 심해집니다.탐험 중간에 피로도가 부족해 무조건 실패할 경로라도 우선 순열로 끝까지 생성하기 때문에 불필요한 연산이 포함됩니다.

📌 결론: 던전의 개수 제한이 $N \le 8$로 매우 작아 두 풀이 모두 통과(Pass)되지만, 백트래킹을 활용하여 실시간으로 조건을 체크하는 피로도sol1 구조가 현업 및 코딩 테스트에서 훨씬 더 지향해야 하는 효율적인 접근법입니다.
